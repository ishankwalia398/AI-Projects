How We Built the Backend

🔄 Ingestion Pipeline

Phase 1 (Current): Hourly Vercel Cron triggers /api/cron → Loader/Parser reads 10 source folders + JIRA MCP (JQL) → Chunker splits by content type → BGE-M3 embedding (dense + sparse) → Vector DB upsert + Metadata store.


Phase 2 (Planned): Real-time webhooks from GitHub, JIRA, Figma → instant ingestion without waiting for hourly job.

  • 10 knowledge source folders
  • Parallel embedding jobs
  • Incremental upserts
  • Metadata tracking for citations
🔍 Retrieval Pipeline

Question → Understand/Rewrite (query expansion) → Parallel Search (Dense vectors + BM25 keyword) → Merge (Reciprocal Rank Fusion) → Cross-Encoder Rerank (BGE-Reranker-v2-M3) → Context Assembly (top 6 chunks) → LLM Answer with inline citations.


Hybrid Search Dense + Sparse Reranking Citations
✂️ Chunking Strategy
  • Code: By function/class (~300-500 tokens)
  • CSV: 10-row groups with header
  • JIRA: Per ticket (split if >800 tokens)
  • PDFs: ~500 tokens by heading
  • Meeting notes: By speaker turn (~600 tokens)
  • Jenkins logs: By build/stage (~400 tokens)
🎯 JIRA MCP Integration

JQL queries run hourly via Vercel Cron. Tickets are normalized, chunked by field (summary, description, comments), embedded with ticket_key metadata for citation.


// Example JQL
project = VWO AND updated >= -7d
ORDER BY updated DESC
📊 Metadata Store

Postgres (Neon) tracks source paths, ticket IDs, line ranges, timestamps for every chunk. Enables precise citation links back to original content.


  • Source file path
  • Line number ranges
  • JIRA ticket keys
  • Last updated timestamp

Query-Time Flow

🔄 How an Answer is Fetched
1
Understand
User question is analyzed and rewritten into multiple search variants to improve recall. Example: "What is KAN-1002?" → ["KAN-1002 bug details", "ticket KAN-1002 status", "JIRA KAN-1002"]
▼
2
Search
Parallel execution:
• Dense Vector Search: BGE-M3 embeddings find semantically similar chunks
• BM25 Keyword Search: Sparse vectors catch exact matches (ticket IDs, method names, exceptions)
Both searches run simultaneously against Qdrant, fetching top 20 candidates each.
▼
3
Rank
Reciprocal Rank Fusion merges dense + sparse results. Then BGE-Reranker-v2-M3 (cross-encoder) rescores all candidates against the original question → top 6 chunks selected.
▼
4
Answer
LLM (Llama 3.3 70B via Groq) receives the 6 chunks as context + original question → generates answer with inline citations. Each citation links to source file, line number, or JIRA ticket.

Deployment & Configuration

🚀 Vercel Deployment

Deployment method: Vercel CLI with token-based auth (non-interactive)

vercel deploy --token=$VERCEL_TOKEN --prod

Environment variables (set via Vercel dashboard or CLI):

QDRANT_URL=https://...cloud.qdrant.io
QDRANT_API_KEY=eyJ...
QDRANT_COLLECTION=qabuddy

GROQ_API_KEY=gsk_...
LLM_BASE_URL=https://api.groq.com/openai/v1
LLM_MODEL=llama-3.3-70b-versatile

POSTGRES_URL=postgresql://neondb_owner:...@ep-...neon.tech/neondb

# Optional for securing cron
CRON_SECRET=your_random_secret

vercel.json configuration:

{
"crons": [
{ "path": "/api/cron", "schedule": "0 * * * *" }
],
"functions": {
"app/api/chat/route.ts": { "maxDuration": 60 },
"app/api/ingest/route.ts": { "maxDuration": 300 },
"app/api/cron/route.ts": { "maxDuration": 300 }
}
}
